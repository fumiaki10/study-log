## React.StrictMode
アプリケーションの潜在的なバグや不適切な実装を、開発環境（developmentモード）で早期に発見・警告してくれるためのツール

画面には描画されず、本番環境のビルドには一切影響を与えない

##　主な動作とチェック内容
* 副作用の検出
レンダー（描画）とエフェクト（副作用）の2回実行を行う。コンポーネント関数やuseEffectのセットアップ処理などが、意図的に2回実行される

⇒console.logが2回表示される、APIが2回飛んでくる

⇒これは何度も同じことをして同じ結果が返ってくるかをテストしている

これにより、予期せぬ副作用が含まれていないかチェックする

* 非推奨のライフサイクルメソッドの警告
将来のReactバージョンで廃止予定のレガシーな機能を使用していないか警告

* 破棄された状態（Ref）の再利用テスト
コンポーネントが再利用される際に、状態が適切にクリーンアップされているか検証


##　レンダーとエフェクト
１．レンダー(再描画)
**見た目を決める計算処理**
* 役割：コンポーネント（関数）を実行、どんなHTML（JSX）を出力するかをReactが計算

* ルール：この処理は「純粋」であること。計算の途中でデータを書き換えたり、APIと通信してはいけない
```
例、return<div>{props.name}</div>
```
のような見た目の組み立て

２． エフェクト（副作用）
画面に映った
**「後」に実行される、外部とのやり取りの**
こと

* 役割：主にuseEffectを使って、画面の計算以外の処理（副作用）を行う

* タイミング：レンダー（再描画）が全て終わり、実際にブラウザに画面が描画された直後に動く

例、サーバーからのデータ取得（API通信）、タイマーの設定、手動でのDOM操作
## render関数
昔はclassから書いていて、ここから先が出力パーツであるとReactに伝えるためにrender()という名前のメソッドの中に処理を書くという厳格なルールがあった

今は無くても動く

```jsx
// 昔の書き方（クラスコンポーネント：renderが必須）
class MyComponent extends React.Component {
  render() {
    return <h1>こんにちは！</h1>;
  }
}
```


###　render関数内
* 状態の不整合と無限ループ
レンダー関数内で状態（state）を変更したり、グローバル変数を書き換えたりすると、それが次のレンダーを引き起こし、無限ループ、予期せぬ画面のちらつきが発生する

* データの重複や消失
レンダーのたびに配列に要素をpushしたり、ＡＰＩリクエストを直接発行したりすると、ＵＩに同じデータが何重にも表示されたり、リクエストが多重送信される原因にもなる

* タイマーやイベントリスナーの多重登録
レンダー中に、setIntervalなどを直接実行すると、コンポーネントが再描画されるたびに新しいタイマーが起動し、処理が爆発的に増える原因にもなる
